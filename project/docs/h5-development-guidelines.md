# H5 设计与开发规范

## 开发原则

- 新 H5 或 H5 原型优先读取 `project/docs/h5-framework-overview.md`、`project/docs/patterns/h5-*.md`、最接近的 `project/h5/src/views/` 实际页面和 `project/h5/src/views/demo/`。
- 页面基于 `project/h5/` 增量开发，不重新搭建 Vue/Vite/Vant 工程。
- 优先使用项目组件；没有匹配组件时先组合 Vant 组件，再新增项目组件。
- 业务页面按功能目录拆分，复杂页面将展示区、表单区、选择器和反馈浮层拆成独立组件。
- 新路由显式登记，不恢复全目录自动扫描，避免历史业务路由被意外装载。
- 默认无认证。需要认证时必须先明确后端协议、宿主 App、Token、身份和异常跳转规则。

## UI 设计规范

- 以 375px 移动端视口设计，同时验证 320px、390px、414px。
- 页面背景使用浅灰，内容区使用白色；主色沿用 `--van-blue: #2D65F0`。
- 卡片圆角控制在 8px，避免过度卡片化和装饰性渐变。
- 页面标题 18-20px，分区标题 16px，正文 14px，辅助信息 12px。
- 表单优先使用 `van-cell-group` 与 `van-field`；标签、输入、错误提示和禁用状态必须完整。
- 主操作使用蓝色实心按钮，次操作使用默认按钮，危险操作使用红色文字或描边。
- 选择器和底部操作使用项目现有底部浮层组件；详情或筛选可使用右侧 Popup。
- 图片上传必须支持选择、预览、删除、失败和数量限制；原型阶段可使用本地读取，不得假装上传成功到服务端。
- 设备能力必须说明 HTTPS、权限和宿主限制，并提供取消路径。
- 所有可见按钮都必须可点击，加载、空、错误、禁用和成功状态按需求覆盖。

## 代码规范

- 使用 `<script setup lang="ts">` 和 Composition API。
- 组件 props、事件和数据模型使用 TypeScript。
- 业务状态使用 `ref`、`reactive`、`computed`，不在模板中堆叠复杂逻辑。
- 页面只负责组合，重复交互下沉到 `src/components/` 或页面 `components/`。
- 使用 `@/` 别名；Vant 组件由自动解析插件按需引入。
- 样式使用 Less，页面样式优先 `scoped`；共享变量和覆盖放入 `src/assets/less/`。
- 网络请求走 `src/utils/http/`，开发阶段接口不可用时同步维护 mock。
- 不在通用组件中写死业务文案、接口地址、权限角色或学校名称。

## 原型规范

- H5 原型应优先复用 `src/views/` 中最接近的实际业务页面结构，再使用 `/demo` 补充组件能力，不用桌面后管布局缩放代替。
- 原型按真实页面拆分路由，菜单、底部导航、详情、表单和弹窗需实际可切换。
- 字段、按钮、状态和异常必须来自需求；Demo 中的示例数据不能被当成固定业务规则。
- 需求没有 H5 设计信息时，以本规范和 Vant 官方交互为基线，并把推断项标记为待确认。
- 原型使用 `.codex/skills/snowy-h5-app-designer/assets/h5-prototype-framework/` 多文件骨架，必须保留标注运行时。
- 自动标注必须绑定具体需求节点；用户可对任意业务节点新增标注，标注按全局/页面作用域隔离并持久化。
- 原型生成前先输出逐页蓝图，生成后运行 H5 蓝图和原型校验脚本。

## 验证

```powershell
cd project/h5
pnpm lint
pnpm build
```

还需在浏览器验证：

- `/` 正确进入 `/demo`。
- 刷新 `/demo` 不触发登录或认证请求。
- 组件交互无控制台错误。
- 弹层可关闭，表单可输入，选择器可确认，上传可本地预览。
- 移动端视口无横向溢出和内容遮挡。
