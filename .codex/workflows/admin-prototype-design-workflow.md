# Snowy 后管拟真原型设计工作流

本工作流用于生成后管、后台、管理端、运营端的 HTML 低保真原型。目标不是画通用线框，而是基于 `project/snowy-admin-web/` 的真实 Snowy 管理端框架，生成和框架高度一致、可供开发评审的拟真原型。

本工作流不绑定具体业务。页面范围、菜单、字段、状态、权限和验收标准必须来自 `docs/requirements/` 下的需求文档、标书拆解结果、PRD 或最小需求说明。

## Demo 参考

后管原型设计前，必须先打开并参考本 Demo：

```text
.codex/skills/snowy-admin-prototype-designer/assets/prototype-demo-framework/index.html
```

该 Demo 由 `.codex/skills/snowy-admin-prototype-designer` 提供。后管原型生成、审查或重做时必须先使用该 skill，并读取其蓝图模板和验收清单。该 Demo 使用 CDN 引入 Vue、Ant Design Vue 和 dayjs，展示 Snowy 后管布局壳、标准 CRUD、左树 + 表格、菜单资源树、抽屉表单、详情抽屉、删除确认、导入弹窗、权限态、空态、异常态、节点标注、页面需求说明和另存为。实际业务后管原型必须先复制该 Demo 的 `index.html` 作为模板，再替换系统标题、Logo、菜单、字段、数据和流程；不得从空白 HTML、通用后台模板或纯静态页面重新绘制。框架参考清单和原型需求覆盖矩阵是交付材料，不应伪装成业务菜单或业务页面。

Demo 中的标注只用于展示“编号气泡、节点选择、详情弹窗、编辑和持久化”的交互形式，不是固定业务规则。实际业务原型的标注内容必须来自当前需求文档、标书原文、PRD、最小需求说明或已校验的页面蓝图；不得复制 Demo 的日期范围、状态枚举、上传限制、删除规则等示例内容。标注只标重要或需要特别说明的具体需求点，例如字段校验、枚举值、状态含义、权限差异、敏感数据脱敏、风险操作、异常处理和业务规则；不得用“筛选区”“表格区”“操作区”等泛泛说明替代具体需求。页面底部不得生成 `annotation-card` 或需求说明列表；当前页面整体需求从顶部工具栏的“页面需求”入口打开右侧抽屉查看和编辑。

硬性模板门禁：后管原型 HTML 必须保留 Demo 版本标识、`prototypeMeta`、Vue + Ant Design Vue CDN、`.snowy-sider`、`.snowy-header`、`.tabs-row`、查询卡片、工具栏、`a-table`、`a-drawer`、`a-modal`、顶部标注工具栏、节点标注与本地持久化、页面需求抽屉和“另存为”能力。图片上传和组件预设是可复用能力：需求涉及对应字段时必须复用，需求不涉及时不得把“组件预设”作为业务菜单或页面保留。缺少必需关键结构时，视为未按 Demo 生成，必须重做，不得进入 UI 设计、技术设计或开发。

硬性设计规则门禁：后管原型不仅要使用 Demo 模板，还必须符合 Demo 的原型设计规则。至少包括：菜单层级来自需求；系统标题、Logo 和简称来自需求或标记待确认；查询区、工具栏、表格、分页、操作列、抽屉/弹窗按 Demo 结构组织；字段按语义展示为图片、头像、文件、标签、开关、徽标、进度、长文本省略、树形选择、更多操作等合适形态；业务状态和启停操作按 Snowy 习惯拆分为标签列和开关列；上传字段可选择本地文件/图片并支持回显、预览和移除；所有可点击元素必须有对应交互；页面不得出现开发实现提示或无关教学说明。任一规则不满足时，视为原型不合格，必须重做。

硬性需求蓝图门禁：后管原型 HTML 生成前，必须先产出“需求到原型页面蓝图”。蓝图必须逐个独立菜单/页面列出需求来源、原始需求摘录、原子需求清单、页面类型、同步字段、展示字段、筛选字段、详情字段、新增字段、编辑字段、状态字段、敏感字段、操作按钮、状态/异常、权限、字段展示形态和点击交互。每个字段、按钮、状态和权限必须标记来源：`需求明确`、`框架惯例`、`待确认` 或 `不适用`。没有蓝图、蓝图字段不来自需求、蓝图未通过 `validate_admin_blueprint.py`、或多个业务页面共用同一套通用字段时，不得生成 HTML。

硬性禁止万能 CRUD：不得用一个通用表格配置或一个通用新增/编辑/详情抽屉覆盖多个业务页面。每个独立页面必须拥有和需求匹配的专属查询区、表格列、详情内容、表单字段、操作列和状态流。确实可复用组件结构时，也必须用页面专属配置驱动，且覆盖矩阵能追溯到具体需求编号和页面蓝图。

Demo 和实际原型页面里不要放流程说明横幅、教学提示、无关统计卡片或为了展示而添加的无关功能。页面内容只保留框架结构、业务必需信息和当前功能需要的操作。能使用图标表达的工具操作必须使用图标按钮，并通过 `title` 或悬浮说明表达含义，不用“刷、密、列”等文字占位。

实际业务原型的系统标题、左上角 Logo、简称和浏览器标题必须来自需求文档、标书、项目名称或用户确认项，不得沿用 Demo 默认值或固定写 `Snowy Admin`。需求未提供时，在原型和覆盖矩阵中标记 `待确认: 系统名称/Logo`。

Demo 和实际原型中所有看起来可点击的元素都必须能点击，并实现对应的原型功能或明确反馈，包括菜单、页签、查询、重置、展开/收起、新增、编辑、详情、删除、导入、导出、审核、授权、分页、勾选、状态开关、表格工具图标、顶部工具图标、抽屉保存和弹窗确认。

表格和详情内容必须按字段类型真实展示，不能用文字代替字段形态。例如：封面图/头像/附件预览必须展示图片缩略图或真实占位图；状态字段用标签或开关；布尔字段用开关或是否标签；时间字段用日期时间；排序字段用数字；权限字段用角色、操作范围或按钮组；缺失图片用空图占位，不写“已上传/未上传”代替图片。

用户可见界面不得出现开发实现提示、框架组件名、内部变量名、接口/mock 说明、权限函数、选择器或组件调用名，例如 `s-table`、`xn-*`、`hasPerm`、`data-*`、`toolConfig`、`触发`、`缓存刷新` 等。技术说明只能写在框架参考清单、覆盖矩阵或设计说明中，不得出现在业务页面、表单字段、表格内容、弹窗、Toast 或按钮文案里。

原型生成时应优先复用 Demo 中的预设组件和展示规则：图片卡片上传、导入拖拽上传、头像、封面图、文件类型、状态标签、开关、徽标、进度条、长文本省略、树形选择、更多操作、确认弹窗和详情/编辑抽屉。需求字段能匹配这些类型时，不得退化成纯文本占位。

## 使用时机

当需求涉及后管、后台、管理端或运营端页面，并且未明确跳过原型时，Product Agent 必须先执行本工作流，再输出后管低保真 HTML 原型。

如果同时涉及 H5/移动端，后管原型和 H5/移动端原型必须分开文件输出。本工作流只约束后管原型；H5/移动端原型必须按移动端框架另行设计。

## 阶段 1：读取 Snowy 框架基线

必须读取：

```text
project/docs/patterns/frontend-crud-pattern.md
project/docs/patterns/feature-capability-map.md
project/docs/patterns/module-map.md
project/snowy-admin-web/src/layout/**
project/snowy-admin-web/src/router/**
project/snowy-admin-web/src/views/sys/**
project/snowy-admin-web/src/views/biz/**
project/snowy-admin-web/src/components/**
```

至少选择 2-3 个真实页面作为视觉和交互参考：

```text
project/snowy-admin-web/src/views/sys/user/index.vue
project/snowy-admin-web/src/views/sys/role/index.vue
project/snowy-admin-web/src/views/sys/resource/menu/index.vue
project/snowy-admin-web/src/views/sys/org/index.vue
project/snowy-admin-web/src/views/biz/notice/index.vue
```

按需求类型选择最相近参考页：

| 需求类型 | 优先参考 |
| --- | --- |
| 普通单表 CRUD | `biz/notice` 或 `sys/position` |
| 人员、组织、角色、权限 | `sys/user`、`sys/org`、`sys/role` |
| 菜单、模块、按钮、资源 | `sys/resource/menu`、`sys/resource/button` |
| 树形组织或层级数据 | `sys/org`、`sys/resource/menu` |
| 导入导出 | `sys/user/impExp.vue` |
| 授权、分配、范围配置 | `sys/role/grantResourceForm.vue`、`sys/user/grantPermissionForm.vue` |

产出“框架参考清单”：

```text
功能/页面:
参考框架文件:
复用布局:
复用组件:
CRUD 形式:
弹窗/抽屉形式:
权限按钮模式:
状态展示模式:
```

没有框架参考清单，不得进入原型绘制。

## 阶段 2：菜单和页面映射

根据需求章节、PRD 或最小需求说明，先建立后管菜单树，再映射页面。

必须输出：

```text
一级菜单:
二级菜单:
三级菜单:
路由路径:
权限标识:
可见角色:
页面类型:
是否独立页面:
对应需求编号:
```

规则：

- 父级章节只做分组时，标记为 `菜单分组`。
- 子章节有独立列表、查询、表单、详情、审核、配置或状态流时，标记为 `独立页面`。
- 不确定是否成菜单时，标记 `待确认: 是否作为菜单`，但原型中仍保留候选入口。
- 原型左侧菜单必须体现该菜单树，不得只把功能放进卡片。

## 阶段 3：按框架确定 CRUD 形式

每个后管页面必须先判断页面类型：

| 页面类型 | Snowy 原型形式 |
| --- | --- |
| 标准 CRUD | 查询表单 + 工具栏 + `s-table` 风格表格 + 分页 + 操作列 + 新增/编辑抽屉 |
| 树形 CRUD | 左树/上级选择 + 表格或树表 + 新增/编辑抽屉 |
| 配置管理 | 查询/分类 + 表格 + 状态开关 + 配置弹窗 |
| 内容管理 | 查询 + 表格 + 状态标签 + 图片/富文本字段 + 发布/下架操作 |
| 导入导出 | 工具栏导入/导出 + 导入弹窗 + 结果反馈 |
| 授权分配 | 列表 + 授权弹窗/抽屉 + 树形勾选/穿梭/范围配置 |
| 审核流程 | 列表 + 状态标签 + 详情抽屉 + 审核弹窗 + 驳回原因 |
| 只读查询 | 查询表单 + 表格 + 详情抽屉 + 导出 |

标准 CRUD 页面必须包含：

- 查询区：关键词、状态、分类、时间范围、组织/归属等需求字段。
- 工具栏：新增、批量删除、导入、导出、刷新、更多操作，按需求和权限决定。
- 表格区：序号、核心字段、状态、排序、创建/更新时间、操作列。
- 操作列：查看、编辑、删除、启用/停用、审核、更多，按需求决定。
- 分页区：总数、页码、每页条数。
- 新增/编辑：优先按 Snowy 习惯使用抽屉或表单容器；可在设计说明中标注参考组件，但不得写进原型可见文案。
- 删除/批量删除：确认弹窗。
- 权限态：无权限时隐藏或禁用对应按钮，并在覆盖矩阵记录。
- 状态：正常、空数据、加载、接口异常、校验失败、删除确认、保存中、保存成功。

## 阶段 4：需求到原型页面蓝图

绘制 HTML 前必须先输出逐页蓝图。蓝图不是说明文字，而是 Product Agent 生成原型的结构化输入；Orchestrator 必须按蓝图验收 HTML。

每个独立页面必须填写：

```text
需求编号:
页面名称:
菜单路径:
路由路径:
权限标识:
可见角色:
页面类型:
参考 Snowy 页面:
原始需求摘录:
原子需求清单:
同步字段:
展示字段:
筛选字段:
查询字段:
表格字段:
详情字段:
新增字段:
编辑字段:
导入/导出字段:
状态字段:
敏感字段:
操作按钮:
状态/异常:
权限差异:
字段展示形态:
点击交互:
空态/加载/错误态:
覆盖矩阵行:
```

字段蓝图规则：

- 查询字段、表格字段、详情字段、新增字段、编辑字段必须分别列出，不得只写“同表格字段”或“通用表单”。
- 必须保留每个页面对应的原始需求摘录；不得只写摘要。
- 必须把需求拆成原子项，至少覆盖适用的同步字段、展示字段、筛选字段、详情字段、状态字段、敏感字段、操作、异常和权限。
- 字段必须来自需求、标书、PRD 或最小需求说明；需求未明确时写 `待确认`，不得擅自用 Demo 字段填充。
- 每个字段、按钮、状态和权限必须标记来源：`需求明确`、`框架惯例`、`待确认` 或 `不适用`。
- 禁止用 `等状态`、`多条件筛选`、`相关字段`、`已列出`、`同新增`、`与上一页一致` 等压缩写法代替字段枚举。
- 只读查询、同步展示类页面不得默认添加新增、编辑、删除、导入、导出；如框架或运营需要这些操作，必须标记 `待确认` 并写明原因。
- 新增、编辑、详情三类表单可以共用组件结构，但字段、只读态、必填态、默认值、校验和按钮必须按页面单独配置。
- 每个表格字段必须声明展示形态：文本、脱敏文本、图片、头像、附件、金额、数字、日期时间、状态标签、启停开关、进度、徽标、树形路径、长文本省略、操作组。
- 每个操作按钮必须声明点击后的原型行为：页面切换、打开抽屉、打开弹窗、带入当前行数据、修改当前行状态、删除当前行、筛选列表、清空条件、上传回显、预览、审批提交或错误反馈。
- 覆盖矩阵中的 `已覆盖` 必须能对应到蓝图中的页面、字段和交互；不能只在矩阵里自称覆盖。
- 覆盖矩阵必须逐需求、逐页面、逐字段或逐操作写清楚，禁止用 `ADM-S-001~020` 这类范围行代替逐项覆盖。

禁止：

- 多个页面共用 `标题/名称、状态、排序、备注` 这类万能表单。
- 只生成菜单和列表，不生成对应新增/编辑/详情/审核/授权/配置流程。
- 只用一套自动规则按字段名猜展示形态，未在蓝图中说明。
- 覆盖矩阵标记 `已覆盖`，但 HTML 中没有对应页面、字段、操作或交互。

蓝图完成后必须运行：

```text
python .codex/skills/snowy-admin-prototype-designer/scripts/validate_admin_blueprint.py docs/design/<需求ID>-admin-page-blueprint.md
```

如某页面有用户强调的必需字段，必须通过 `--must-contain` 追加校验。蓝图校验出现 `FAIL` 时不得生成 HTML；出现会导致字段遗漏、万能 CRUD 或虚假覆盖的 `WARN` 时也必须先修正。

## 阶段 5：绘制 Snowy 后管布局壳拟真原型

后管 HTML 原型必须先搭建 Snowy 管理端壳：

- 左侧菜单栏，包含一级/二级/三级菜单和当前选中态。
- 顶部栏，包含面包屑、标签页或页签、用户区/操作区的框架占位。
- 内容区，按 Snowy 页面密度设计，不使用营销页、卡片墙或移动端布局。
- 页面标题、查询表单、工具栏、表格、分页、抽屉/弹窗必须在视觉上接近 Snowy 管理端。
- 所有可点击元素必须绑定交互，至少体现页面切换、弹窗/抽屉打开关闭、状态切换、勾选、分页、查询/重置和保存反馈。
- 表格单元格必须按字段类型展示真实内容，图片字段必须展示缩略图或空图占位，不得用“已上传/未上传”等文字替代。
- 上传字段必须可点击选择本地文件或图片，并在原型中回显、预览和移除，图片上传优先参考 Snowy 的卡片式图片上传形态。
- 颜色、间距、按钮、标签、表格密度应参考框架，而不是重新设计一套风格。

禁止：

- 只画“功能导航卡片”。
- 放置流程说明横幅、教学提示、无关统计卡片或与当前需求无关的功能。
- 图标工具按钮使用文字占位。
- 可点击元素没有交互。
- 表格内容和表头语义不匹配。
- 图片、头像、附件等字段用文字代替真实预览。
- 只画少量代表页面。
- 用通用后台模板替代 Snowy 布局。
- 把 CRUD 画成移动端列表或营销卡片。
- 把新增/编辑/详情等关键流程只写文字不画弹窗/抽屉。
- 使用一个万能 `currentConfig`、万能表单或万能抽屉让所有页面显示同一套字段。

## 阶段 6：覆盖全部需求细节

原型必须覆盖需求集合中的全部后管功能点。功能很多时，应拆成多个 HTML 文件或多个页面区域，不得压缩成概览。

必须覆盖：

- 每个菜单入口。
- 每个独立页面。
- 每个查询条件。
- 每个表格字段。
- 每个新增/编辑/详情字段。
- 每个操作按钮。
- 每个可点击元素的交互反馈。
- 每个表格字段的真实展示形态。
- 每个状态标签、开关或状态流。
- 每个权限差异。
- 每个异常状态和空状态。
- 每个导入、导出、审核、启停、删除、批量操作等特殊流程。

必须输出“原型需求覆盖矩阵”：

| 需求编号 | 功能点 | 页面/弹窗/抽屉 | 蓝图条目 | 字段 | 操作 | 状态/异常 | 权限 | 覆盖状态 | 说明 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |

覆盖状态只能是：

- `已覆盖`
- `待确认`
- `不适用`

不得出现未解释的 `未覆盖`。如果确实不能覆盖，必须写明原因和下一步确认问题。

## 阶段 7：验收门禁

进入 UI/Figma、技术方案或开发前，Orchestrator 必须检查：

- 已读取 `frontend-crud-pattern.md` 和真实 Snowy 页面。
- 已输出框架参考清单。
- 已输出需求到原型页面蓝图，且每个独立页面都有专属查询字段、表格字段、详情字段、新增字段、编辑字段、操作按钮、状态/异常、权限和字段展示形态。
- 蓝图已通过 `validate_admin_blueprint.py`；没有 `等状态`、`多条件筛选`、`同新增`、范围覆盖行或无来源字段。
- 蓝图字段均可追溯到需求、标书、PRD 或最小需求说明；待确认字段已标记，未用 Demo 字段冒充业务字段。
- 后管 HTML 原型包含 Snowy 后管布局壳，且标题、Logo、系统名已按具体需求替换或标记待确认。
- 后管 HTML 原型是从 `.codex/skills/snowy-admin-prototype-designer/assets/prototype-demo-framework/index.html` 复制后改造，不是从空白 HTML 重新编写。

- 后管 HTML 原型逐页实现蓝图，不存在多个业务页面共用同一个万能表单、万能抽屉或万能字段集的情况。
