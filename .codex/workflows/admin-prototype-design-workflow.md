# Snowy 后管拟真原型设计工作流

本工作流用于生成后管、后台、管理端、运营端的 HTML 低保真原型。目标不是画通用线框，而是基于 `project/snowy-admin-web/` 的真实 Snowy 管理端框架，生成和框架高度一致、可供开发评审的拟真原型。

本工作流不绑定具体业务。页面范围、菜单、字段、状态、权限和验收标准必须来自 `docs/requirements/` 下的需求文档、标书拆解结果、PRD 或最小需求说明。

## Demo 参考

后管原型设计前，必须先打开并参考本 Demo：

```text
docs/design/prototype-demo-framework/index.html
```

该 Demo 使用 CDN 引入 Vue、Ant Design Vue 和 dayjs，展示 Snowy 后管布局壳、标准 CRUD、左树 + 表格、菜单资源树、抽屉表单、详情抽屉、删除确认、导入弹窗、权限态、空态和异常态。实际业务原型可以复制该目录后替换菜单、字段、数据和流程，但不得降低这些框架元素和覆盖要求。框架参考清单和原型需求覆盖矩阵是交付材料，不应伪装成业务菜单或业务页面。

Demo 和实际原型页面里不要放流程说明横幅、教学提示、无关统计卡片或为了展示而添加的无关功能。页面内容只保留框架结构、业务必需信息和当前功能需要的操作。能使用图标表达的工具操作必须使用图标按钮，并通过 `title` 或悬浮说明表达含义，不用“刷、密、列”等文字占位。

实际业务原型的系统标题、左上角 Logo、简称和浏览器标题必须来自需求文档、标书、项目名称或用户确认项，不得沿用 Demo 默认值或固定写 `Snowy Admin`。需求未提供时，在原型和覆盖矩阵中标记 `待确认: 系统名称/Logo`。

Demo 和实际原型中所有看起来可点击的元素都必须能点击，并实现对应的原型功能或明确反馈，包括菜单、页签、查询、重置、展开/收起、新增、编辑、详情、删除、导入、导出、审核、授权、分页、勾选、状态开关、表格工具图标、顶部工具图标、抽屉保存和弹窗确认。

表格和详情内容必须按字段类型真实展示，不能用文字代替字段形态。例如：封面图/头像/附件预览必须展示图片缩略图或真实占位图；状态字段用标签或开关；布尔字段用开关或是否标签；时间字段用日期时间；排序字段用数字；权限字段用角色、操作范围或按钮组；缺失图片用空图占位，不写“已上传/未上传”代替图片。

用户可见界面不得出现开发实现提示、框架组件名、内部变量名、接口/mock 说明、权限函数、选择器或组件调用名，例如 `s-table`、`xn-*`、`hasPerm`、`data-*`、`toolConfig`、`触发`、`缓存刷新` 等。技术说明只能写在框架参考清单、覆盖矩阵或设计说明中，不得出现在业务页面、表单字段、表格内容、弹窗、Toast 或按钮文案里。

原型生成时应优先复用 Demo 中的预设组件和展示规则：图片卡片上传、导入拖拽上传、头像、封面图、文件类型、状态标签、开关、徽标、进度条、长文本省略、树形选择、更多操作、确认弹窗和详情/编辑抽屉。需求字段能匹配这些类型时，不得退化成纯文本占位。

## 使用时机

当需求涉及后管、后台、管理端或运营端页面，并且未明确跳过原型时，Product Agent 必须先执行本工作流，再输出后管低保真 HTML 原型。

如果同时涉及 H5/移动端，后管原型和 H5/移动端原型必须分开文件输出。本工作流只约束后管原型；H5/移动端原型必须按移动端框架另行设计。

## 阶段 1：读取 Snowy 框架基线

必须读取：

```text
project/docs/patterns/frontend-crud-pattern.md
project/docs/patterns/feature-capability-map.md
project/docs/patterns/module-map.md
project/snowy-admin-web/src/layout/**
project/snowy-admin-web/src/router/**
project/snowy-admin-web/src/views/sys/**
project/snowy-admin-web/src/views/biz/**
project/snowy-admin-web/src/components/**
```

至少选择 2-3 个真实页面作为视觉和交互参考：

```text
project/snowy-admin-web/src/views/sys/user/index.vue
project/snowy-admin-web/src/views/sys/role/index.vue
project/snowy-admin-web/src/views/sys/resource/menu/index.vue
project/snowy-admin-web/src/views/sys/org/index.vue
project/snowy-admin-web/src/views/biz/notice/index.vue
```

按需求类型选择最相近参考页：

| 需求类型 | 优先参考 |
| --- | --- |
| 普通单表 CRUD | `biz/notice` 或 `sys/position` |
| 人员、组织、角色、权限 | `sys/user`、`sys/org`、`sys/role` |
| 菜单、模块、按钮、资源 | `sys/resource/menu`、`sys/resource/button` |
| 树形组织或层级数据 | `sys/org`、`sys/resource/menu` |
| 导入导出 | `sys/user/impExp.vue` |
| 授权、分配、范围配置 | `sys/role/grantResourceForm.vue`、`sys/user/grantPermissionForm.vue` |

产出“框架参考清单”：

```text
功能/页面:
参考框架文件:
复用布局:
复用组件:
CRUD 形式:
弹窗/抽屉形式:
权限按钮模式:
状态展示模式:
```

没有框架参考清单，不得进入原型绘制。

## 阶段 2：菜单和页面映射

根据需求章节、PRD 或最小需求说明，先建立后管菜单树，再映射页面。

必须输出：

```text
一级菜单:
二级菜单:
三级菜单:
路由路径:
权限标识:
可见角色:
页面类型:
是否独立页面:
对应需求编号:
```

规则：

- 父级章节只做分组时，标记为 `菜单分组`。
- 子章节有独立列表、查询、表单、详情、审核、配置或状态流时，标记为 `独立页面`。
- 不确定是否成菜单时，标记 `待确认: 是否作为菜单`，但原型中仍保留候选入口。
- 原型左侧菜单必须体现该菜单树，不得只把功能放进卡片。

## 阶段 3：按框架确定 CRUD 形式

每个后管页面必须先判断页面类型：

| 页面类型 | Snowy 原型形式 |
| --- | --- |
| 标准 CRUD | 查询表单 + 工具栏 + `s-table` 风格表格 + 分页 + 操作列 + 新增/编辑抽屉 |
| 树形 CRUD | 左树/上级选择 + 表格或树表 + 新增/编辑抽屉 |
| 配置管理 | 查询/分类 + 表格 + 状态开关 + 配置弹窗 |
| 内容管理 | 查询 + 表格 + 状态标签 + 图片/富文本字段 + 发布/下架操作 |
| 导入导出 | 工具栏导入/导出 + 导入弹窗 + 结果反馈 |
| 授权分配 | 列表 + 授权弹窗/抽屉 + 树形勾选/穿梭/范围配置 |
| 审核流程 | 列表 + 状态标签 + 详情抽屉 + 审核弹窗 + 驳回原因 |
| 只读查询 | 查询表单 + 表格 + 详情抽屉 + 导出 |

标准 CRUD 页面必须包含：

- 查询区：关键词、状态、分类、时间范围、组织/归属等需求字段。
- 工具栏：新增、批量删除、导入、导出、刷新、更多操作，按需求和权限决定。
- 表格区：序号、核心字段、状态、排序、创建/更新时间、操作列。
- 操作列：查看、编辑、删除、启用/停用、审核、更多，按需求决定。
- 分页区：总数、页码、每页条数。
- 新增/编辑：优先按 Snowy 习惯使用抽屉或表单容器；可在设计说明中标注参考组件，但不得写进原型可见文案。
- 删除/批量删除：确认弹窗。
- 权限态：无权限时隐藏或禁用对应按钮，并在覆盖矩阵记录。
- 状态：正常、空数据、加载、接口异常、校验失败、删除确认、保存中、保存成功。

## 阶段 4：绘制 Snowy 后管布局壳拟真原型

后管 HTML 原型必须先搭建 Snowy 管理端壳：

- 左侧菜单栏，包含一级/二级/三级菜单和当前选中态。
- 顶部栏，包含面包屑、标签页或页签、用户区/操作区的框架占位。
- 内容区，按 Snowy 页面密度设计，不使用营销页、卡片墙或移动端布局。
- 页面标题、查询表单、工具栏、表格、分页、抽屉/弹窗必须在视觉上接近 Snowy 管理端。
- 所有可点击元素必须绑定交互，至少体现页面切换、弹窗/抽屉打开关闭、状态切换、勾选、分页、查询/重置和保存反馈。
- 表格单元格必须按字段类型展示真实内容，图片字段必须展示缩略图或空图占位，不得用“已上传/未上传”等文字替代。
- 上传字段必须可点击选择本地文件或图片，并在原型中回显、预览和移除，图片上传优先参考 Snowy 的卡片式图片上传形态。
- 颜色、间距、按钮、标签、表格密度应参考框架，而不是重新设计一套风格。

禁止：

- 只画“功能导航卡片”。
- 放置流程说明横幅、教学提示、无关统计卡片或与当前需求无关的功能。
- 图标工具按钮使用文字占位。
- 可点击元素没有交互。
- 表格内容和表头语义不匹配。
- 图片、头像、附件等字段用文字代替真实预览。
- 只画少量代表页面。
- 用通用后台模板替代 Snowy 布局。
- 把 CRUD 画成移动端列表或营销卡片。
- 把新增/编辑/详情等关键流程只写文字不画弹窗/抽屉。

## 阶段 5：覆盖全部需求细节

原型必须覆盖需求集合中的全部后管功能点。功能很多时，应拆成多个 HTML 文件或多个页面区域，不得压缩成概览。

必须覆盖：

- 每个菜单入口。
- 每个独立页面。
- 每个查询条件。
- 每个表格字段。
- 每个新增/编辑/详情字段。
- 每个操作按钮。
- 每个可点击元素的交互反馈。
- 每个表格字段的真实展示形态。
- 每个状态标签、开关或状态流。
- 每个权限差异。
- 每个异常状态和空状态。
- 每个导入、导出、审核、启停、删除、批量操作等特殊流程。

必须输出“原型需求覆盖矩阵”：

| 需求编号 | 功能点 | 页面/弹窗/抽屉 | 字段 | 操作 | 状态/异常 | 权限 | 覆盖状态 | 说明 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |

覆盖状态只能是：

- `已覆盖`
- `待确认`
- `不适用`

不得出现未解释的 `未覆盖`。如果确实不能覆盖，必须写明原因和下一步确认问题。

## 阶段 6：验收门禁

进入 UI/Figma、技术方案或开发前，Orchestrator 必须检查：

- 已读取 `frontend-crud-pattern.md` 和真实 Snowy 页面。
- 已输出框架参考清单。
- 后管 HTML 原型包含 Snowy 后管布局壳，且标题、Logo、系统名已按具体需求替换或标记待确认。
- 左侧菜单树和需求菜单一致。
- 每个独立功能都有页面或弹窗/抽屉。
- CRUD 形式和 Snowy 框架一致。
- 查询、工具栏、表格、分页、操作列、抽屉/弹窗、确认弹窗齐全。
- 状态、空态、异常、权限态齐全。
- 已输出原型需求覆盖矩阵。
- 覆盖矩阵没有未解释的缺口。
- 如同时有 H5/移动端，已分开输出 H5 原型文件。

任一项不满足，状态为 `阻塞`，不得进入下一阶段。
