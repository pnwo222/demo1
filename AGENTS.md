# AGENTS.md

本项目是一个外卖平台，技术方向为：

- 前端：Vue 3 H5，面向移动端用户体验。
- 后端：Java 服务端，面向订单、商户、商品、购物车、支付、配送和用户体系。

所有 Codex Agent 在修改本项目之前，都应先阅读本文件，并遵守这里的协作规则。

## 项目目标

构建一个可迭代的外卖平台开发流程，覆盖：

用户浏览商家 -> 选择商品 -> 加入购物车 -> 下单 -> 支付 -> 商家接单 -> 配送 -> 完成/取消/退款

平台必须重点保证：

- 下单链路稳定。
- 支付和金额计算正确。
- 库存、购物车、订单状态一致。
- 用户、商家、骑手、平台角色权限清晰。
- 前端 H5 在移动端体验顺滑。
- 后端接口可测试、可监控、可回滚。
- PRD 必须输出为可打开的 HTML 文档，便于产品、设计、开发和测试统一评审。
- UI 设计必须连接 Figma 生成可编辑设计图，并优先使用 `.codex/skills/ui-ux-pro-max` 建立设计系统。

## 基本原则

- 不直接在主分支上开发，功能改动使用独立 branch 或 worktree。
- 不让开发 Agent 自己给自己放行，必须经过 Review、CI 和人工审批。
- 不在订单状态、金额规则、库存规则、权限规则不清时直接写代码。
- 前端优先复用现有 Vue 组件、路由、状态管理和样式规范。
- 后端优先遵循现有 Java 分层架构、接口规范、异常处理和事务规范。
- 涉及支付、退款、库存、订单状态、用户数据、商家资金、删除和批量操作的改动必须重点审查。
- 所有功能都必须有验收标准、测试结果和残余风险说明。

## 外卖平台重点规则

### 金额和支付

- 金额计算必须以后端为准，前端只能展示和预估。
- 支付金额、优惠金额、配送费、包装费、退款金额必须有清晰来源。
- 不能用浮点数直接计算金额，应使用整数分或 Java BigDecimal。
- 支付回调必须校验签名、金额、订单号和幂等性。
- 支付、退款、取消订单必须记录审计日志。

### 订单状态

订单状态必须设计为明确状态机，不能随意跳转。

示例状态：

- 待支付
- 已支付
- 商家已接单
- 制作中
- 待配送
- 配送中
- 已完成
- 已取消
- 退款中
- 已退款

每次状态变化必须明确：

- 谁触发
- 允许从哪个状态变更到哪个状态
- 是否影响库存
- 是否影响支付或退款
- 是否需要通知用户、商家或骑手
- 是否记录审计日志

### 库存和购物车

- 购物车价格展示可以实时计算，但最终下单价格必须重新校验。
- 下单时必须校验商品是否上架、库存是否足够、商家是否营业、配送地址是否可送达。
- 库存扣减、订单创建、支付确认之间必须有清晰一致性策略。
- 防止重复下单、重复支付、重复退款。

### 权限和数据隔离

- 用户只能查看和操作自己的订单。
- 商家只能管理自己的门店、商品和订单。
- 骑手只能查看分配给自己的配送任务。
- 平台运营需要按角色控制可见范围。
- 后端必须做权限校验，不能只依赖前端隐藏按钮。

## 标准开发流程

0. Orchestrator Agent 先声明当前阶段、调用 Agent、输入、输出、验收标准和下一阶段。
1. Orchestrator 调用 Product Agent 生成外卖业务 PRD、验收标准和 HTML 版 PRD。
2. Orchestrator 调用 Design Agent 生成低保真原型；确认后再使用 `.codex/skills/ui-ux-pro-max` 生成设计系统，并连接 Figma 直接生成可编辑设计图。
3. Orchestrator 调用 Architect Agent 明确模块边界、订单状态机、API、数据模型、安全模型和可运维性。
4. Orchestrator 调用 Data Agent 细化数据库模型、migration、索引、回滚和数据一致性策略。
5. Orchestrator 按用户价值拆 feature slice。
6. Frontend、Backend、Data、QA 等 Agent 在独立 branch 或 worktree 并行开发。
6. 本地运行必要检查后提交 PR。
7. Reviewer、Security、QA Agent 做审查。
8. CI 运行 lint、typecheck、test、build、安全扫描。
9. 人工负责人审批后合并。
10. 预发验证、灰度发布、全量发布。
11. 发布后监控订单、支付、接口错误率和用户反馈。

## Definition of Done

一个功能完成必须满足：

- PRD 或需求说明已确认。
- HTML 版 PRD 已生成并可打开。
- H5 交互状态和移动端适配已明确。
- Figma 设计图已生成或更新，且设计规则来自 `ui-ux-pro-max` 设计系统。
- 技术方案已明确模块边界、接口、数据、安全和运维要求。
- 订单、支付、库存、权限等核心规则已写清。
- 代码实现遵循现有项目规范。
- 必要测试已补充并通过。
- CI 检查通过。
- Coverage 未低于主分支基线。
- SAST、SCA、Secret Scan、DAST 按风险等级通过。
- Review 中的 P0/P1 问题已修复。
- 涉及发布的变更有灰度和回滚方案。
- 验收标准逐条通过或记录未通过项。

- Secret Scan
- DAST

## 审查优先级

- P0：支付错误、越权操作、重复扣款、订单数据损坏、生产不可用，必须修复。
- P1：影响下单、购物车、支付、退款、配送、商家接单或核心体验的问题，合并前应修复。
- P2：体验或可维护性改进，不一定阻塞上线。
