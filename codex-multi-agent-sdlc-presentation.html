Operating Model

推荐工作模式：需求集合驱动，总控编排，专业 Agent 并行交付

总控 Agent

读取 docs/requirements/ 下的全部需求文档。
拆 feature slice，生成 task graph 和依赖 DAG。
按 wave 分配 Agent、branch/worktree 和集成策略。
维护状态板，推动 CI、Review、QA 和发布闭环。

专业 Agent

只修改 files_allowed 范围内的文件。
在独立 branch 或 worktree 开发。
必须产出验证结果和风险说明。
不绕过 CI、Review 和人工审批。

Task Breakdown

阶段四：任务拆分，先切 Slice，再生成可并行执行的任务图

Slice

按用户价值切最小业务闭环，明确输入、输出和验收。

Task Graph

每个任务包含 owner、范围、产物、检查命令和风险等级。

DAG

识别依赖关系，避免 API、数据和 UI 契约互相猜测。

Wave

按可并行层次推进：契约、主体实现、联调、门禁。

Queue

通过 integration branch 或 PR queue 统一集成。

每个任务都要有 files_allowed，越界修改先回报 Orchestrator；共享核心文件时改为串行或指定单一 owner。

任务拆解09 / 15
Auto Dispatch

阶段五：自动分配与并行开发，用 wave 控制速度和风险

自动分配规则

Wave 0：产品、设计、架构补齐规格。
Wave 1：API 契约、数据底座、QA 测试计划。
Wave 2：前端、后端、数据、测试用例并行实现。
Wave 3：联调、修复、smoke、回归。

隔离与集成

不同目录可独立 branch，同仓多模块优先 worktree。
公共契约由单一 owner 维护，消费方不抢写。
低风险任务进 integration branch，高风险任务单独 PR。
CI 通过后再进入最终 Review 和人工审批。

阶段七：审查和集成，不让开发 Agent 自己给自己放行

Reviewer重点

Performance Reviewer慢查询、重复请求、缓存、并发
Maintainability Reviewer复杂度、重复代码、模块边界
QA Reviewer缺失测试、验收标准覆盖
Codex CLI: /review 或： 请以代码审查模式检查当前分支相对 main 的 diff。 重点关注真实 bug、权限风险、性能风险和缺失测试。 按 P0 / P1 / P2 输出必须修改项。

Release Phase

阶段八：部署发布，用灰度和回滚控制生产风险

1

CI 构建

测试、构建、镜像、安全扫描全部通过。

2

预发

接近生产的环境完成 UAT 和联调。

3

Feature Flag

功能开关控制暴露范围。

4

Canary

小流量发布，观察错误率和核心指标。

5

全量

逐步扩大流量，确认无异常后完成。

6

回滚

异常时按预案回滚代码、配置和数据。

Implementation Plan

从 0 到 1 的落地路线：先轻量，再自动化

第 1 步

把业务需求拆入 docs/requirements/，流程和角色保持通用。

第 2 步

跑通 PRD、低保真 HTML、Figma、技术设计和数据设计。

第 3 步

启用自动分配流程，输出 task graph、DAG、wave 和 owner。

第 4 步

接入 worktree、integration branch、CI、安全扫描和 Review。

持续

根据阻塞、冲突、返工和线上指标改进 workflow 与门禁。

最佳实践：用 Agent 提速，用流程兜底，用数据复盘。
