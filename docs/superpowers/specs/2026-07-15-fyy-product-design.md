# 纺院一卡通 Product 阶段设计规格

## 1. 文档目的

本规格用于确认“宁波市高校一卡通专区”Product 阶段的交付范围、后管信息架构、页面行为、高风险规则和验收门禁。后续 PRD、后管页面蓝图与低保真原型必须以本规格和 `docs/requirements/纺院需求.md` 为共同输入；业务事实仍以标书原子需求为准。

## 2. 已确认决策

| 决策项 | 确认结果 |
| --- | --- |
| 开发模式 | 高风险严格模式 |
| Product 交付方案 | 统一交付包 |
| 后管原型组织 | 一个 Snowy 后管 HTML，通过菜单切换学校端、管理端、接口中心和 PAM 管理页面 |
| H5 框架 | 不采用当前未跟踪的 `project/h5/` 作为正式基线 |
| H5 设计 | 本阶段明确跳过，不生成 H5 页面蓝图或 H5 原型 |
| H5 需求处理 | 继续保留在 PRD 的延期范围和需求追踪中，不视为已实现或已完成设计 |
| 纯硬件规格 | 排除采购参数，仅保留设备接入、认证、状态监控和远程配置的软件约束 |

## 3. Product 交付物

本阶段在开发者完成书面规格确认后生成：

1. 中文 Markdown PRD。
2. 与 Markdown 同步的单文件 HTML PRD。
3. 后管框架参考清单。
4. 后管菜单与页面映射。
5. 逐独立页面的需求到原型页面蓝图。
6. 基于 Snowy Demo 模板生成的单文件后管低保真 HTML 原型。
7. 逐需求、逐页面、逐字段和逐操作的原型需求覆盖矩阵。

本阶段不生成 H5 原型、Figma 稿、技术架构、数据库方案或业务代码。

## 4. 后管信息架构

### 4.1 学校端

- 业务首页。
- 师生管理：学生管理、教职工管理、历史学生管理。
- 内容管理：首页 Banner、应用上线、首页模块、公告、文章、活动。
- 社保卡申领记录。
- 设备状态监控。
- 信息查询：访客记录、通行记录、消费记录、图书借阅记录。
- 用户管理。

### 4.2 管理端

- 系统首页。
- 组织架构：学校管理、用户管理。
- 权限管理：角色管理、模块管理、菜单管理。
- 日志管理：访问日志、操作日志、外部调用日志。
- 宁波一卡通后管对接。
- 审核中心。
- 系统监控。

### 4.3 接口中心

- 学院数据初始化。
- 人员信息管理。
- 人员关键信息核验。
- 人员信息同步。
- 电子社保码认证。
- 实体社保卡认证。
- 移动支付交易信息推送。
- 实体社保卡交易信息接收。

### 4.4 PAM 管理

- 组织架构与人员管理。
- 掌静脉识别算法特征库。
- 设备分组管理。
- 人员分组管理。
- 用户通行实时同步。
- 通行记录与用户信息导入导出。
- 设备运维监测与远程配置。
- 日志与分级权限管理。

菜单分组和权限约束类需求作为信息架构或横切规则处理；只有具备独立查询、表单、详情、配置、审核或监控内容的功能才生成独立业务页面。

## 5. 页面和操作规则

### 5.1 同步只读页面

学生、教职工、消费、借阅、通行、社保卡记录等来源于外部系统或数据中台的页面默认采用“查询表单 + 表格 + 详情抽屉”的只读形式。标书未明确新增、编辑、删除、导入或导出的页面不得擅自增加这些操作。

### 5.2 内容运营页面

Banner、应用、首页模块、公告、文章和活动按照标书明确的新增、编辑、删除、发布、推荐、展示或审核能力设计。字段和状态必须逐页配置，禁止多个页面共用万能表单或万能抽屉。

### 5.3 组织与权限页面

学校、部门、用户、角色、模块、菜单复用 Snowy 组织树、用户列表、角色授权、菜单资源树和权限按钮模式。授权范围必须体现角色、资源、组织范围和操作权限，前端按钮可见性不能替代后端权限校验。

### 5.4 设备与 PAM 页面

设备在线状态、异常状态、通行同步、远程配置、人员分组、设备分组和分级子账户分别设计专属页面或弹窗。远程配置、授权变更、批量导入导出等高风险操作必须二次确认并呈现操作结果。

## 6. 角色与权限

| 角色 | 主要范围 |
| --- | --- |
| 平台管理员 | 管理端全局配置、学校组织、角色权限、接口与监控 |
| 学校管理员 | 本校师生、内容、记录、设备和用户数据 |
| 内容运营人员 | Banner、应用、首页模块、公告、文章和活动 |
| 审核人员 | 公告、内容、访客或其他明确审核任务 |
| 安全审计人员 | 访问、操作、接口、设备通信和权限变更日志 |
| 设备运维人员 | 设备状态、异常、分组、远程信息和配置 |
| PAM 分级子账户 | 按管理范围与查看、编辑、授权权限访问 PAM 资源 |
| 接口调用方 | 仅访问已授权接口，不进入普通业务菜单 |

## 7. 敏感数据与高风险规则

- 身份证号、手机号、社保卡号在列表中脱敏展示；详情和导出受权限控制。
- 掌静脉特征按生物识别敏感数据处理，原型不展示真实模板内容，只显示受控的特征状态、数量或标识。
- 敏感数据导出必须提示用途与风险，并记录导出人、时间、条件和结果。
- 人员批量导入必须展示校验、重复项、异常项和结果反馈。
- 设备远程配置、权限授权、删除和批量操作必须二次确认并记录审计信息。
- 移动支付和实体卡交易接口在后续技术方案中必须落实签名、幂等、金额校验、重复推送处理、失败重试和对账机制；Product 原型体现处理状态与异常入口，不虚构未给出的接口字段。

## 8. 页面状态与异常

每个适用页面必须覆盖：

- 初始加载与加载中。
- 正常数据与分页。
- 空数据。
- 查询无结果。
- 接口或同步异常。
- 无权限。
- 表单校验失败。
- 保存中、保存成功和保存失败。
- 高风险操作确认。
- 导入校验结果或导出反馈。
- 设备离线、设备异常或外部调用失败等业务异常。

## 9. 原型设计约束

- 后管 HTML 必须从 `.codex/skills/snowy-admin-prototype-designer/assets/prototype-demo-framework/index.html` 复制后改造。
- 必须保留 Snowy 布局壳、查询区、工具栏、表格、分页、抽屉、弹窗、顶部标注工具、节点标注、本地持久化、页面需求抽屉和另存为能力。
- 页面字段按图片、头像、附件、状态标签、启停开关、金额、日期、脱敏文本、权限范围、长文本等语义展示。
- 所有可见可点击控件必须产生与文案一致的原型反馈。
- 不在业务界面展示开发变量、组件名、接口 Mock 说明或工作流提示。
- 不生成 H5 原型，也不把 H5 页面混入后管原型。

## 10. 验收与验证

1. PRD 可追溯到 62 个原子需求；H5-001 至 H5-013 标记为“需求保留、设计延期”，不得标记为本阶段原型已覆盖。
2. 每个独立后管页面拥有专属查询、表格、详情、表单、操作、状态、权限和交互定义。
3. 页面蓝图通过 `validate_admin_blueprint.py`，不存在压缩字段、万能 CRUD 或虚假覆盖。
4. 后管原型通过 `validate_admin_prototype.py`，保留 Demo 强制结构。
5. 使用运行时检查打开原型，确认无脚本错误、空白页或核心点击故障。
6. 覆盖矩阵中的每个“已覆盖”项均能定位到蓝图和 HTML；H5 项明确记录本阶段不适用及跳过原因。
7. 敏感数据、批量操作、权限、交易接口和设备远程控制均进入高风险审查清单。

## 11. 明确不在本阶段范围

- H5 页面设计与 H5 原型。
- Figma 高保真 UI。
- 接口协议字段、签名算法和错误码的最终技术定义。
- 数据库实体、表结构、索引和 migration。
- 前后端业务代码和真实接口联调。
- 硬件采购、数量、材质、电气和安装规格。

上述内容不是遗漏：H5 由开发者明确延期；技术、数据和实现内容由后续阶段在各自门禁下完成。
