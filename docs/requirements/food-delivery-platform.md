# 外卖平台需求说明

## 项目定位

本项目是一个外卖平台，当前技术方向为：

- 前端：Vue 3 H5，面向移动端用户体验。
- 后端：Java 服务端，面向订单、商户、商品、购物车、支付、配送和用户体系。

## 核心链路

```text
用户浏览商家 -> 选择商品 -> 加入购物车 -> 下单 -> 支付 -> 商家接单 -> 配送 -> 完成/取消/退款
```

## 业务角色

| 角色 | 说明 |
| --- | --- |
| 用户 | 浏览商家、点餐、下单、支付、查看订单、申请退款 |
| 商家 | 管理门店、商品、库存、接单、出餐 |
| 骑手 | 接收配送任务、更新配送状态 |
| 平台运营 | 审核商家、管理类目、处理订单异常 |
| 管理员 | 配置系统、处理权限、安全和运维 |

## 核心业务模块

- 用户登录与地址管理
- 商家列表与商家详情
- 商品类目与 SKU
- 购物车
- 优惠券、满减、配送费、包装费
- 订单创建、支付、取消、退款
- 商家接单、出餐
- 配送状态
- 评价
- 后台运营管理
- 审计日志和异常订单处理

## 重点质量目标

- 下单链路稳定。
- 支付和金额计算正确。
- 库存、购物车、订单状态一致。
- 用户、商家、骑手、平台角色权限清晰。
- 前端 H5 在移动端体验顺滑。
- 后端接口可测试、可监控、可回滚。

## 金额和支付规则

- 金额计算必须以后端为准，前端只能展示和预估。
- 支付金额、优惠金额、配送费、包装费、退款金额必须有清晰来源。
- 不能用浮点数直接计算金额，应使用整数分或 Java BigDecimal。
- 支付回调必须校验签名、金额、订单号和幂等性。
- 支付、退款、取消订单必须记录审计日志。

## 订单状态规则

订单状态必须设计为明确状态机，不能随意跳转。

示例状态：

- 待支付
- 已支付
- 商家已接单
- 制作中
- 待配送
- 配送中
- 已完成
- 已取消
- 退款中
- 已退款

每次状态变化必须明确：

- 谁触发
- 允许从哪个状态变更到哪个状态
- 是否影响库存
- 是否影响支付或退款
- 是否需要通知用户、商家或骑手
- 是否记录审计日志

## 库存和购物车规则

- 购物车价格展示可以实时计算，但最终下单价格必须重新校验。
- 下单时必须校验商品是否上架、库存是否足够、商家是否营业、配送地址是否可送达。
- 库存扣减、订单创建、支付确认之间必须有清晰一致性策略。
- 防止重复下单、重复支付、重复退款。

## 权限和数据隔离

- 用户只能查看和操作自己的订单。
- 商家只能管理自己的门店、商品和订单。
- 骑手只能查看分配给自己的配送任务。
- 平台运营需要按角色控制可见范围。
- 后端必须做权限校验，不能只依赖前端隐藏按钮。

- Unit Test
- Build
- Coverage

- Unit Test

- Build

## 审查优先级

- P0：支付错误、越权操作、重复扣款、订单数据损坏、生产不可用，必须修复。
- P1：影响下单、购物车、支付、退款、配送、商家接单或核心体验的问题，合并前应修复。
- P2：体验或可维护性改进，不一定阻塞上线。
