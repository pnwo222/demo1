## 2. 背景与目标

基于宁波一卡通 APP 建设高校一卡通学校专区，实现社保卡实体卡、社保码电子码、校园卡、掌静脉、人脸等身份认证与支付场景融合。系统同时建设移动端学校专区、学校端后管、平台管理端后管、校园类接口和 PAM 人员设备协同管理能力。

目标：

- 面向师生和访客提供学校资讯、通知公告、访客预约、校园启航、图书借阅、我的记录、校园卡包等移动端服务。
- 面向学校提供内容发布、师生数据、社保卡申领记录、设备状态、访客/通行/消费/借阅记录和用户汇聚管理。
- 面向平台提供多学校组织、用户、角色、菜单、日志、对接监控和数据看板。
- 面向系统集成提供人员初始化、人员同步、身份核验、社保码/卡认证、交易推送和交易接收接口。
- 面向 PAM 提供人员、生物信息、设备、授权、通行、远程配置和日志管理。

| 角色 | 范围 | 核心诉求 |

| 内容管理员 | 本校内容 | 管理 banner、公告、文章、活动、首页模块和应用 |

## 4. 核心业务流程

1. 学校和组织初始化：平台管理员创建学校、部门、角色、菜单和用户。
2. 人员数据同步：接口完成学院、学生、教职工和历史学生数据初始化、核验、同步和状态回传。
3. 内容发布：内容管理员维护 banner、应用、首页模块、公告、文章、活动，经审核后同步到 APP 学校专区。
4. H5 使用：师生/访客进入学校专区，查看资讯、公告、地图、招生咨询、图书、我的记录，发起访客预约或校园启航流程。
5. 认证与记录：社保卡/码、校园卡、掌静脉、人脸在门禁、图书、消费等场景认证，记录同步到后管查询。
6. PAM 运维：设备、人员、分组、授权、通行、远程配置和日志持续管理。
7. 审计与监控：访问日志、操作日志、外部调用日志、接口失败和设备告警进入审计闭环。

## 7. 状态和业务规则

| 类别 | 规则 |
| --- | --- |
| 数据隔离 | 学校端仅能访问本校数据；平台端按角色数据范围访问。 |
| 敏感信息 | 身份证、手机号、社保卡号默认脱敏；导出需审批、留痕。 |
| 内容发布 | 内容支持草稿、待审核、已发布、已下架、已驳回、已过期状态。 |
| 交易记录 | 金额、流水号、支付类型、付款方式、支付时间必须准确；重复流水幂等。 |
| 认证接口 | 社保码认证需校验登录态、身份、码有效期、场景权限和防重放。 |
| 远程设备配置 | 下发前二次确认，保存回滚点，失败可重试并记录通信日志。 |
| PAM 生物信息 | 生物特征模板按高敏感数据处理，最小权限访问并完整审计。 |
| 日志 | 登录、操作、外部调用、设备通信、导出和远程配置必须留痕。 |

## 8. 非功能需求

- 安全：Sa-Token 权限、数据范围、敏感字段脱敏/加密、导出审批、接口验签、日志审计。
- 性能：常用列表分页查询；设备状态和接口日志支持时间范围检索；高频通行记录需索引和归档策略。
- 可用性：接口失败可重试；设备离线补传；导入有错误行报告；导出有审批状态反馈。
- 可运维：接口调用日志、失败告警、设备在线状态、同步任务状态、远程配置回执。
- 兼容性：H5 需适配宁波一卡通 APP 容器；后管复用 Snowy 管理端组件和权限体系。
