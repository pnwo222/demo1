背景与目标

纺院高校一卡通专区以社保卡、社保码、校园卡、掌静脉、人脸等身份载体为基础，建设 APP 学校专区、学校端后管、管理端后管、校园类接口和 PAM 人员设备管理能力。目标是实现校园身份认证、内容发布、师生信息同步、通行/消费/借阅记录查询、设备状态监控、外部接口对接和审计追踪。

功能范围

H5/APP 学校专区

  • 首页：banner、校园资讯、通知公告、特色应用入口。
  • 学校信息：学校简介、专业介绍、校园地图、招生咨询。
  • 通知资讯：公告列表/详情、资讯列表/详情、资讯推荐到首页 banner。
  • 访客预约：姓名、身份证号、手机号、访问时间、通行方式、车牌、拜访人、缘由、备注、审核状态。
  • 校园启航：新生账号激活、社保卡办理预约、招生录取数据比对。
  • 图书借阅：新书推荐、图书检索、借阅排行。
  • 我的：个人资料、访客预约记录、消费记录、图书借阅记录、门禁记录、校园卡账本、社保卡信息、校园码。
  • 一卡通 APP 对接：服务消息、我的卡包。

PAM 人员设备管理

  • 组织架构管理、人员管理、掌静脉生物信息录入。
  • 掌静脉识别算法特征库，容量 5 万。
  • 设备分组、人员分组、授权管理。
  • 用户通行实时同步，通行记录导出，用户信息导入导出。
  • 设备在线/离线、异常检测、远程配置、远程信息查询。
  • 用户操作日志、设备通信日志。
  • 授权子账户和多级分级权限。

核心流程

内容发布流程

  • 内容运营创建 banner、应用、模块、公告、文章或活动。
  • 系统校验标题、图片、展示期、可见身份、跳转链接和排序。
  • 需要审核的内容进入待审核。
  • 审核通过后按展示期、展示状态和可见身份在 H5/APP 展示。
  • 内容过期、停用或被驳回时不展示。

人员同步和身份核验流程

  • 学校数据中台推送或接口拉取学生、教职工、历史学生数据。
  • 系统做格式校验、身份标识去重、异常数据拦截。
  • 对接持卡、绑卡、金融账户激活和社保码/卡认证结果。
  • 后管展示注册、绑卡、省内持卡、卡规范、金融账户激活状态。
  • H5 侧按登录态展示个人资料和记录。

交易接收流程

  • 接收移动支付或实体社保卡交易。
  • 验签、校验金额、校验流水号唯一性。
  • 重复流水标记为重复处理，不重复入账。
  • 失败记录进入外部调用日志和待处理队列。

设备远程配置流程

  • 运维选择设备和配置项。
  • 系统提示影响范围并要求审批确认。
  • 通过后下发配置并保留配置快照。
  • 下发失败可重试，必要时按快照回滚。

状态与业务规则

对象 状态 规则
内容 草稿、待审核、已发布、已驳回、已下线、已过期 待审核和驳回内容不展示；超过结束时间自动过期
应用 可用、停用、展示、隐藏 停用或隐藏时 H5 不展示
预约 待审核、已通过、未通过、已取消、已过期 仅已通过可用于入校
人员持卡 未注册、已注册、未绑卡、已绑卡、省内持卡、金融账户未激活、金融账户已激活 状态来自外部同步或核验
设备 在线、离线、故障、告警、维护中 离线和告警需突出展示
交易 已接收、校验失败、重复流水、已入账、待处理 流水号必须唯一幂等
远程配置 待审批、已通过、已下发、失败、已回滚 高风险操作必须审计

非功能需求

  • 权限：菜单、按钮、API 权限必须一致；高风险导出和远程配置使用最小权限。
  • 安全：身份证号、手机号、社保卡号、支付流水、掌静脉特征、门禁记录脱敏展示。
  • 审计：导出、远程配置、权限变更、人员同步、交易接收必须记录操作日志。
  • 性能：大表列表分页查询；人员、通行、消费、日志需要索引和归档策略。
  • 可用性：外部接口失败要可重试、可定位、可人工处理。
  • 兼容：后管复用 Snowy 管理端；H5 框架待确认。
