Key Message

多 Agent 不是“一个 AI 写完整个系统”，而是把成熟 SDLC 拆给专业角色协作

人类负责方向、取舍、审批和验收；Agent 负责检索、草稿、实现、验证、修复和总结。

团队结构04 / 15
Operating Model

推荐工作模式：总控 Agent 编排，专业 Agent 并行，PR 汇总

总控 Agent

读取 PRD、设计稿、代码库和历史规范。
拆分 feature slice，分配给专业 Agent。
汇总结果、发现冲突、输出集成计划。
推动审查、测试、发布和验收闭环。

专业 Agent

只负责清晰边界内的任务。
在独立 branch 或 worktree 开发。
必须产出验证结果和风险说明。
不绕过 CI、Review 和人工审批。

技术方案08 / 15
Task Breakdown

阶段四：任务拆分，按用户价值切 Slice，而不是按岗位切大包

Slice 1

数据模型、基础 API、空页面、权限框架。先搭最小闭环。

Slice 2

列表查询、筛选、分页、空状态、错误状态。

Slice 3

创建、编辑、校验、重复提交、乐观反馈。

Slice 4

权限、审计日志、异常处理、安全扫描。

Slice 5

E2E、预发、灰度、验收报告、发布说明。

每个 Slice 都应该能独立评审、独立测试，最好能独立发布或通过 Feature Flag 控制。

测试策略11 / 15
Code Review

阶段七：审查，不让 Agent 自己给自己放行

Reviewer重点
Bug Reviewer真实缺陷、边界条件、兼容性
Security Reviewer权限绕过、数据泄漏、注入、依赖风险
Performance Reviewer慢查询、重复请求、缓存、并发
Maintainability Reviewer复杂度、重复代码、模块边界
QA Reviewer缺失测试、验收标准覆盖
Codex CLI: /review 或： 请以代码审查模式检查当前分支相对 main 的 diff。 重点关注真实 bug、权限风险、性能风险和缺失测试。 按 P0 / P1 / P2 输出必须修改项。
审查机制12 / 15
Release Phase

阶段八：部署发布，用灰度和回滚控制生产风险

1

CI 构建

测试、构建、镜像、安全扫描全部通过。

2

预发

接近生产的环境完成 UAT 和联调。

3

Feature Flag

功能开关控制暴露范围。

4

Canary

小流量发布，观察错误率和核心指标。

5

全量

逐步扩大流量，确认无异常后完成。

6

回滚

异常时按预案回滚代码、配置和数据。

业务验收14 / 15
Implementation Plan

从 0 到 1 的落地路线：先轻量，再自动化

第 1 周

建立 AGENTS.md、PR 模板、测试命令和基础工作规范。

第 2 周

配置 product、architect、frontend、backend、qa 五类 agent。

第 3 周

用一个真实小需求跑完整流程，沉淀提示词和检查表。

第 4 周

接入 Figma、GitHub、CI、安全扫描和自动 Review。

持续

根据事故、返工和指标改进 agent 规则与质量门禁。

最佳实践：用 Agent 提速，用流程兜底，用数据复盘。
